# PRD — D-Agent (보안 로그 분석 서비스)

### 1. 문서 정보

- **프로젝트명**: D-Agent
- **문서 목적**: 제품/기술 방향 정렬 및 구현 범위 정의
- **대상 독자**: 개발자(백엔드/프론트), DevOps, 운영/보안 담당자, PO

---

### 2. 배경 및 기획 의도

보안/운영 로그는 양이 많고 맥락이 분산되어 있어, 인력 기반 수동 분석만으로는 **탐지 지연**, **우선순위 판단 오류**, **재발 방지 조치 누락**이 발생하기 쉽습니다.  
D-Agent는 LLM(Gemini)을 활용해 로그를 신속히 요약하고, 위험 신호와 대응 방향을 제시하여 **초기 트리아지 시간을 단축**하는 것을 목표로 합니다.

---

### 3. 목표(Goals) / 비목표(Non-Goals)

#### 목표

- **빠른 트리아지**: 원인 후보, 의심 지점, 즉시 조치를 제안하여 분석 시간 단축
- **일관된 분석 품질**: 담당자 숙련도에 따른 편차 감소
- **운영 친화성**: `/docs` 기반 문서화, Docker 기반 배포, EC2 운영 가능
- **진화 가능성**: “수동 분석 → DB 자동 저장”으로 기능 확장 가능한 기반 확보

#### 비목표(현 시점)

- SIEM/EDR의 완전 대체
- 자동 격리/차단 같은 강제적 액션의 자동 실행
- 모든 로그 포맷에 대한 완벽한 파서 제공(초기에는 텍스트 기반 입력 중심)

---

### 4. 타겟 유저 및 사용 시나리오

#### 타겟 유저

- **보안 담당자/관제 요원**: 경보/이상 로그를 빠르게 요약하고 대응 가이드 확보
- **SRE/운영자**: 장애/에러 로그에서 원인 후보와 조치 순서를 빠르게 얻기
- **개발자**: 배포/인증/권한 관련 이슈의 단서 탐색 및 재현 포인트 확보

#### 대표 시나리오

- 사용자가 서비스에서 수집한 로그(또는 알림 메시지)를 붙여 넣고 분석 요청
- 시스템은 Gemini를 호출하여 결과(요약, 의심 포인트, 대응 권고)를 반환
- 운영자는 결과를 기반으로 추가 조사(로그 확대 수집, 권한 점검, IP 차단 등)를 수행
- (발전 방향) 분석 요청/응답을 DB에 자동 저장하여 히스토리 검색/대시보드로 확장

---

### 5. 현재 제공 기능(As-Is)

#### API

- **루트 안내**
  - `GET /`
  - 서버 상태, 프로젝트 정보, `/docs` 링크, 주요 엔드포인트 경로 안내(JSON)
- **로그 분석**
  - `POST /api/v1/agent/analyze`
  - 입력 로그를 분석하여 요약/인사이트를 반환
- **문서**
  - Swagger UI: `GET /docs`

#### 운영/배포

- **기술 스택(운영/배포)**
  - Python 3.10+
  - FastAPI
  - Docker / Docker Compose
  - AWS EC2
  - Supabase(PostgreSQL)
- **Docker 기반 실행**
  - `backend/Dockerfile` 기반 컨테이너 실행
  - `docker-compose.yml`로 포트 매핑(`8000:8000`) 및 `.env` 주입
- **AWS EC2 배포 전제**
  - EC2에서 Docker로 백엔드 컨테이너 운영 가능한 형태

### 6. 핵심 기능 요구사항(To-Be)

#### 6.1 로그 분석(고도화)

- 입력 로그에서 다음을 포함한 결과 제공
  - 이벤트 요약(무슨 일이 일어났는지)
  - 위험도/우선순위 판단 근거
  - 가능한 원인 후보(가설) 및 확인 방법(추가로 봐야 할 로그/지표)
  - 즉시 조치(단기) 및 재발 방지(중장기) 권고

#### 6.2 “수동 분석 → DB 자동 저장” 발전(진행 중 방향성)

현재는 요청/응답이 API 호출 단위로만 처리되는 흐름이 중심이며, 다음 단계로 **분석 히스토리를 DB에 자동 저장**하는 방향으로 확장합니다.

- 분석 요청/응답 자동 저장(히스토리/감사 추적/재현성 확보)
- 조회 API 제공(최근 분석 목록, 기간/태그 필터)
- 재분석(동일 입력 대비 결과 비교) 및 품질 개선 루프 구축

예시 데이터 필드(초안):

- `id`: UUID(Primary Key)
- `log_hash`: 원문 로그 해시(SHA-256 등). 중복 분석 방지/캐시 키로 활용
- `risk_score`: 정규화 점수(예: 0~100). 우선순위/대시보드 정렬 기준
- `analysis_result`: JSONB(모델 출력의 구조화 결과). 요약/근거/권고/태그 포함
- `created_at`: timestamp with time zone(UTC)
- `source`: 입력 채널(예: manual, api, collector)
- `model`: 사용 모델 식별자/버전
- `meta`: JSONB(요청 컨텍스트, 파서 결과, 마스킹 여부 등 확장 메타데이터)

### 7. 비기능 요구사항(NFR)

- **보안**
  - API 키/시크릿은 코드에 하드코딩 금지. 환경 변수 주입을 기본으로 하며, 운영 환경은 Secrets Manager(또는 동등 개념)로 중앙 관리/회전(Rotation) 가능해야 함
  - 로그 데이터는 민감정보(토큰/세션/이메일/IP 등) 포함 가능. 저장 전/전송 전 **마스킹/필터링 정책** 필요(정규식/룰 기반 + 예외 목록)
  - 운영 환경에서 인바운드 포트 접근은 최소화(허용 IP 제한 권장)
- **가용성**
  - 컨테이너 재시작 정책 등으로 프로세스 다운타임 최소화
- **관측성(Observability)**
  - 요청 실패/예외 발생 시 원인 추적 가능한 로그 확보(향후 구조화 로그로 확장)
- **성능**
  - LLM 호출 시간(네트워크/모델 응답)을 고려한 타임아웃/재시도 정책(향후)
- **사용성**
  - `/docs`를 통해 API 사용법을 즉시 파악 가능
  - 정의되지 않은 경로 요청 시 친절한 404 응답 제공

---

### 8. 제약사항 및 리스크

- **LLM 응답 품질 변동**
  - 동일 입력이라도 결과가 달라질 수 있어, 저장/평가 체계가 중요
- **민감정보 포함 가능성**
  - 로그에 개인정보/토큰이 섞일 수 있어 마스킹/필터링 정책이 필요(향후)
- **비용/쿼터**
  - Gemini 호출은 비용 및 쿼터 제한이 있어, 캐싱/샘플링/요약 저장이 유효(향후)

---

### 9. 성공 지표(Success Metrics)

- 분석 요청 대비 **초기 트리아지 시간 단축**(정성/정량 설문 및 평균 처리시간)
- 재분석/재현을 위한 **히스토리 저장률**(DB 저장 단계 이후)
- 운영자가 “추가로 무엇을 확인해야 하는지”에 대한 안내 만족도

---

### 10. 로드맵(제안)

- **MVP(현재)**: 분석 API + Swagger + Docker/EC2 운영 가능
- **Phase 1**: 분석 히스토리 DB 자동 저장(스키마/마이그레이션/조회 API)
- **Phase 2**: 태깅/검색/대시보드(프론트 연계), 마스킹/정책
- **Phase 3**: 알림 연동(Slack 등), 규칙 기반 프리필터 + LLM 하이브리드, Log Collector(수집 에이전트) 개발을 통한 실시간 파이프라인 자동화
